# Plan: 시나리오 1 — 모든 거터볼 (score 0)

`Game` 클래스에 `roll(pins)`와 `score()` 메서드를 도입한다.
20번의 롤을 모두 거터볼(0핀)로 던졌을 때, `score()`는 `0`을 반환해야 한다.

이번 사이클에서는 **오직 이 시나리오만** 다룬다. 스페어, 스트라이크, 10프레임 보너스 로직은

- `score()`는 저장된 롤들의 합을 반환하는 정도로 최소 구현한다.
  (모든 롤이 0이므로 합계 로직은 이 시나리오를 통과시키는 가장 단순한 코드가 된다.
  스페어/스트라이크 보너스 계산은 이후 사이클에서 테스트가 요구할 때 추가한다.)

# Plan: 시나리오 3 — 스페어 (spare)

## 목표 (Goal)

한 프레임에서 스페어(두 번의 롤 합이 10)가 나온 경우, 그 프레임의 점수는
`10 + 다음 한 번의 롤`이어야 한다.

예: `roll(5), roll(5)` (스페어) `, roll(3)`, 이후 나머지는 모두 거터볼(0)일 때
`score()`는 `10 + 3(보너스) + 3(다음 프레임의 첫 롤 자체 점수) = 16`이어야 한다.

이번 사이클에서는 **오직 스페어 처리**만 다룬다. 스트라이크, 10프레임 보너스 로직은
이후 사이클에서 각각 별도의 Plan으로 다룬다.

## 접근 방식 (Approach)

- 기존의 "전체 롤 합산" 방식(`sum(self.rolls)`)은 스페어 보너스를 반영하지 못하므로,
  `score()`를 프레임 단위로 순회하며 계산하는 방식으로 교체해야 한다.
- 각 프레임을 앞에서부터 순회하면서, 두 롤의 합이 10이면(스페어) 그 프레임 점수 =
  `10 + 다음 한 롤`으로 계산하고, 아니면 두 롤의 단순 합으로 계산한다.
- 아직 스트라이크는 다루지 않으므로, 한 프레임은 항상 두 번의 롤을 소비한다고 가정한다
  (스트라이크 처리는 다음 사이클에서 이 가정을 깨고 확장한다).

## 작성할 실패 테스트 (Failing test to write)

`tests/test_game.py`

```python
def test_one_spare_scores_bonus():
    game = Game()
    game.roll(5)
    game.roll(5)  # 스페어
    game.roll(3)
    for _ in range(17):
        game.roll(0)
    assert game.score() == 16
```

## 예상되는 RED 실패 사유

- 현재 `score()`는 `sum(self.rolls)`이므로 `5+5+3 = 13`을 반환한다.
- 테스트가 기대하는 `16`과 다르므로 `AssertionError: assert 13 == 16`으로 실패해야 한다
  (오타가 아니라 스페어 보너스 로직 부재로 인한 실패).

## 범위 밖 (Out of scope for this cycle)

- 스트라이크 처리

# Plan: 시나리오 4 — 스트라이크 (strike)

## 목표 (Goal)

한 프레임에서 스트라이크(첫 롤이 10)가 나온 경우, 그 프레임은 롤 하나로 끝나고
점수는 `10 + 다음 두 번의 롤`이어야 한다.

예: `roll(10)` (스트라이크) `, roll(3), roll(4)`, 이후 나머지는 모두 거터볼(0)일 때
`score()`는 `(10 + 3 + 4) + (3 + 4) = 24`여야 한다.
(프레임1 = 10+3+4=17, 프레임2 = 3+4=7, 나머지 프레임 = 0)

이번 사이클에서는 **오직 스트라이크 처리**만 다룬다. 10프레임 보너스 롤, 퍼펙트 게임은
이후 사이클에서 각각 별도의 Plan으로 다룬다.

## 접근 방식 (Approach)

- 현재 `score()`는 프레임마다 항상 롤 2개를 소비한다고 가정하는데, 스트라이크는 롤 1개로
  프레임이 끝나므로 이 가정을 깨야 한다.
- 각 프레임 순회 시, 그 프레임의 첫 롤이 10(스트라이크)이면: 프레임 점수 =
  `10 + 다음 두 롤의 합`으로 계산하고, 다음 프레임은 롤 1개만 건너뛴다(포인터를 1 증가).
- 스트라이크가 아니면 기존과 동일하게 스페어/일반 프레임 로직을 그대로 적용하고
  포인터를 2 증가시킨다.

## 작성할 실패 테스트 (Failing test to write)

`tests/test_game.py`

```python
def test_one_strike_scores_bonus():
    game = Game()
    game.roll(10)  # 스트라이크
    game.roll(3)
    game.roll(4)
    for _ in range(16):
        game.roll(0)
    assert game.score() == 24
```

## 예상되는 RED 실패 사유

- 현재 `score()`는 프레임마다 롤 2개를 한 쌍으로 취급하므로, 스트라이크 롤(10)과 다음 롤(3)을
  하나의 프레임으로 잘못 묶어 계산한다 (10+3=13이 스페어 조건(합이 10)에 해당하지 않아
  단순 합산 13으로 처리됨). 그 결과 기대값 `24`와 다른 값을 반환해
  `AssertionError`로 실패해야 한다 (오타가 아니라 스트라이크 로직 부재로 인한 실패).

# Plan: 시나리오 5 — 퍼펙트 게임 (12 스트라이크, score 300)

## 목표 (Goal)

12번 연속 스트라이크(10핀)를 던졌을 때, `score()`는 `300`을 반환해야 한다.

이 시나리오는 동시에 "10프레임 보너스 롤" 규칙도 검증한다: 게임은 정확히 10프레임이며,
10프레임에서 스트라이크/스페어가 나오면 보너스 완성을 위한 추가 롤(최대 3롤까지)이
주어지지만, 그 추가 롤은 별도의 11번째 프레임으로 취급되지 않는다.

## 접근 방식 (Approach)

- 현재 `score()`는 `while i < len(self.rolls) - 1` 조건으로 롤 배열 전체를 소진할 때까지
  순회하며, 프레임 개수를 세지 않는다. 스트라이크가 연속되면 이 조건이 배열 끝을 넘어서는
  인덱스에 접근하게 되어 `IndexError`를 일으킬 수 있다 (10프레임을 넘어서도 계속 순회하기
  때문).
- `score()`를 "정확히 10번" 프레임을 순회하도록 수정한다 (`for _ in range(10)`), 각 프레임
  안에서는 기존 스트라이크/스페어/일반 프레임 판정 로직을 그대로 재사용한다.
- 10프레임을 순회하는 동안, 마지막 프레임에서 스트라이크/스페어가 나와도 추가 롤은 그
  프레임의 보너스 계산에만 쓰이고, 루프는 10번째 프레임에서 종료되므로 11번째 프레임으로
  잘못 처리되지 않는다.

## 작성할 실패 테스트 (Failing test to write)

`tests/test_game.py`

```python

    game = Game()

    assert game.score() == 300
```

## 예상되는 RED 실패 사유

- 현재 `score()`는 프레임 개수를 10개로 제한하지 않고 롤 배열 끝까지 순회하므로, 마지막
  프레임(10번째)의 보너스 롤 이후에도 계속 순회를 시도하다가 존재하지 않는 인덱스에
  접근해 `IndexError: list index out of range`로 실패해야 한다 (오타가 아니라
  10프레임 경계 로직 부재로 인한 실패).
